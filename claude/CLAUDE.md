# 動作要件
- `jq`
- グローバルで外部ツールを使うスクリプト・設定を追加した際はこのセクションを更新する

# コミュニケーション
- 応答は日本語で行う
- 簡潔に回答し自明な説明は省略する
- 指示の意図が不明な場合は推測せず質問する
- 要件・設計・ユーザーの指示の間に矛盾を発見した場合は黙って進めず**積極的に**指摘する
- 要約する時、ユーザーから特に指示が無い場合は日本語で回答する
- 意図が伝わりにくいツール呼び出し（バックグラウンド出力の確認、ログ読み取りなど）の前に「〜を確認します」と一言説明する

# コーディング
- ファイルを新規作成するより、既存ファイルの編集を優先する
- ファイルを修正したらテストを実行する
- シンプルなコードを心がけ、過剰な設計をしない
- 過剰なコメントを書かない
- 命名は分かりやすく、できるだけ短いものを選択する
- Git操作の際は `~/.claude/rules/git.md` を確認し、必ず従うこと。
- git push は**ユーザーからの明示的な指示がない限り実行しない**

# テスト
- テスト項目を考えるときは要件定義書・基本設計書・ユーザーの指示を起点にする。実装ファイルを参照するのはテストの書き方（APIの使い方など）を確認する場合のみ。
- テストケースを書くとき、実装コードを見てそれに合わせたテストを書いてはいけない

# 仕事の流れ
- うまくいかなくなったら一旦立ち止まって計画を練り直す。無理に突き進まない
- タスク完了を報告する前に必ずtestとlintを行う
